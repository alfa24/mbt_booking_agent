# ADR-001: Выбор СУБД для системы бронирования

## Статус
Accepted

## Контекст

Система бронирования загородного жилья требует хранения:
- Пользователей с ролями (арендаторы, арендодатели)
- Домов с характеристиками
- Бронирований с датами и составом гостей
- Тарифов и расчётов стоимости
- Истории заметок о расходниках

Требования к хранилищу:
- Поддержка реляционных данных и целостности
- Гибкость для JSON-полей (состав гостей)
- Простота деплоя на managed-платформах
- Возможность масштабирования без смены технологии

## Рассмотренные варианты

### 1. PostgreSQL

**За:**
- Полноценная реляционная модель с ACID
- Нативная поддержка JSON/JSONB для гибких полей
- Проверенное решение, широкая экосистема
- Простой деплой на Railway, Render, Supabase
- Возможность масштабирования через репликацию

**Против:**
- Требует миграций при изменении схемы
- Не является "serverless" из коробки

### 2. SQLite

**За:**
- Нулевая конфигурация
- Файл базы данных — просто копировать

**Против:**
- Не подходит для concurrent доступа из нескольких сервисов
- Ограниченная масштабируемость
- Нет встроенной репликации

### 3. MongoDB

**За:**
- Гибкая схема документов
- Хорошо подходит для JSON-подобных данных

**Против:**
- Слабая поддержка транзакций на уровне документов
- Необходимость управления отдельным сервисом
- Команда знакома с SQL, не с Mongo

### 4. Firebase / Supabase (managed)

**За:**
- Serverless, автомасштабирование
- Real-time подписки из коробки

**Против:**
- Vendor lock-in
- Firebase — проприетарный, сложно мигрировать
- Supabase — хорош, но PostgreSQL проще для self-hosted

## Решение

Использовать **PostgreSQL** как основную СУБД.

**Обоснование:**
- Универсальность: подходит и для MVP, и для production
- Простота: команда знает SQL, не нужно обучение
- Гибкость: JSONB для неструктурированных данных + реляции для целостности
- Экосистема: легко найти хостинг, инструменты, библиотеки

## Последствия

- Все сервисы (bot, backend, web) работают с одной БД через backend API
- Для кэширования и сессий в будущем добавим Redis
- При масштабировании — репликация чтения, возможно шардирование по домам
