# ADR-002: Выбор backend-фреймворка

## Статус

Accepted

## Контекст

Система бронирования загородного жилья требует backend API, который будет:
- Единой точкой входа для всех клиентов (Telegram-бот, веб-приложение)
- Центром бизнес-логики (бронирования, тарификация, управление домами)
- Интеграционным слоем с внешними сервисами (LLM-провайдер, платёжные системы)

Требования к backend:
- REST API с автоматической документацией
- Async поддержка для интеграций (LLM, Telegram)
- Валидация данных через Pydantic (уже используется в боте)
- Работа с PostgreSQL (выбрано в ADR-001)
- Простота развёртывания и разработки

## Рассмотренные варианты

### 1. FastAPI

**За:**
- Нативная интеграция с Pydantic (та же библиотека, что в боте)
- Async/await из коробки — идеально для LLM-интеграций
- Автоматическая генерация OpenAPI/Swagger документации
- SQLAlchemy 2.0 с полной поддержкой async
- Высокая производительность (основан на Starlette)
- Современный, активно развивается

**Против:**
- Меньше "батареек" из коробки чем Django (нет админки, auth)
- Экосистема меньше, но растёт быстро
- Для сложных бизнес-правил может потребоваться больше архитектурных решений

### 2. Django + Django REST Framework

**За:**
- Много готового функционала из коробки (админка, auth, ORM)
- Зрелая экосистема и большое комьюнити
- DRF мощный для API

**Против:**
- Sync по умолчанию, async поддержка ограничена
- Избыточен для API-only проекта (весь Django ради REST)
- Не нативный Pydantic — своя система сериализации
- Больше "магии", сложнее контроль

### 3. Flask

**За:**
- Минималистичный, гибкий
- Простой для понимания

**Против:**
- Требует ручной настройки многих вещей (валидация, документация)
- Нет нативной async поддержки (Flask 2.0+ есть, но ограничена)
- Устаревает по сравнению с FastAPI для API-проектов

## Решение

Использовать **FastAPI** как backend-фреймворк.

**Стек:**
- FastAPI 0.110+
- SQLAlchemy 2.0 с asyncpg
- Alembic для миграций
- Pydantic v2 для валидации

**Обоснование:**
- Единый стек с ботом (Pydantic) — переиспользование схем
- Native async — критично для LLM-интеграций
- Автодокументация — ускоряет разработку фронтенда
- Современный и производительный
- Подходит для API-first архитектуры

## Последствия

### Положительные
- Единый подход к валидации через Pydantic в боте и backend
- Автоматическая OpenAPI документация доступна по `/docs`
- Высокая производительность за счёт async и Starlette
- Простота тестирования с pytest-asyncio

### Отрицательные
- Нужно реализовывать авторизацию самостоятельно (OAuth2/JWT)
- Нет встроенной админки (можно добавить отдельно, если понадобится)
- Требуется продумать архитектуру для сложной бизнес-логики

## Структура проекта

```
backend/
├── main.py           # Точка входа, создание FastAPI app
├── config.py         # Pydantic-settings для конфигурации
├── dependencies.py   # Dependency injection (DB сессии и т.д.)
├── api/
│   ├── __init__.py   # Роутеры
│   ├── health.py     # Health check endpoint
│   ├── bookings.py   # CRUD бронирований
│   ├── houses.py     # CRUD домов
│   ├── users.py      # CRUD пользователей
│   └── tariffs.py    # CRUD тарифов
├── schemas/
│   ├── __init__.py
│   ├── user.py
│   ├── house.py
│   ├── booking.py
│   ├── tariff.py
│   └── common.py     # ErrorResponse, PaginatedResponse
├── models/
│   ├── __init__.py
│   ├── user.py       # SQLAlchemy модели
│   ├── house.py
│   ├── booking.py
│   └── tariff.py
├── services/
│   ├── __init__.py
│   ├── booking.py    # Бизнес-логика бронирований
│   ├── house.py
│   └── user.py
└── database.py       # Подключение к PostgreSQL
```

## Зависимости

```toml
[project.dependencies]
fastapi = "^0.110.0"
uvicorn = {extras = ["standard"], version = "^0.27.0"}
sqlalchemy = {extras = ["asyncio"], version = "^2.0.0"}
asyncpg = "^0.29.0"
alembic = "^1.13.0"
pydantic = "^2.5.0"
pydantic-settings = "^2.1.0"
```
