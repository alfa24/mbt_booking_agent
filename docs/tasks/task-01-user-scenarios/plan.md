# План: Задача 01 - Сценарии пользователей и требования к данным

## Цель

Определить сценарии использования системы для арендаторов и арендодателей, чтобы заложить основу под будущий frontend и понять, какие данные и связи точно понадобятся.

| № | Сценарий | Описание | Необходимые данные |
|---|----------|----------|-------------------|

### 3. Анализ данных и связей

#### Выводы по сущностям:

**User:**
- id, telegram_id, name, role — базовые поля
- role определяет доступные сценарии (tenant/owner/both)
- Связь: User 1:N House (владение), User 1:N Booking (бронирования)

**House:**
- id, name, description, capacity, owner_id, is_active
- is_active контролирует доступность для бронирования
- Связь: House 1:N Booking, House 1:N ConsumableNote

**Booking:**
- id, house_id, tenant_id, check_in, check_out
- guests_planned (JSON), guests_actual (JSON) — гибкий состав группы
- total_amount — итоговая сумма
- status (pending/confirmed/cancelled/completed)
- Связь: Booking N:1 House, Booking N:1 User

**Tariff:**
- id, name, amount — справочник тарифов
- Используется при расчёте total_amount в Booking

**ConsumableNote:**
- id, house_id, created_by, name, comment, created_at
- Связь: ConsumableNote N:1 House, ConsumableNote N:1 User

**StayRecord (опционально):**
- Может быть отдельной сущностью или частью Booking
- guests_breakdown, notes, recorded_at
