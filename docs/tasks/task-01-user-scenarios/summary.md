# Summary: Задача 01 - Сценарии пользователей и требования к данным

## Что реализовано

### Сценарии арендатора

1. **Просмотр списка домов** — требует House (id, name, description, capacity, is_active)
2. **Просмотр деталей дома** — требует все поля House
3. **Просмотр календаря занятости** — требует Booking (house_id, check_in, check_out, status)
4. **Создание бронирования** — требует Booking + Tariff для расчёта
5. **Просмотр своих бронирований** — требует Booking с фильтром по tenant_id
6. **Отмена бронирования** — требует обновление status в Booking
7. **Фиксация результатов поездки** — требует guests_actual в Booking или StayRecord

### Сценарии арендодателя

1. **Просмотр своих домов** — требует House с фильтром по owner_id
2. **Управление календарём** — требует Booking для блокировки дат
3. **Просмотр всех бронирований** — требует Booking с фильтрами
4. **Управление тарифами** — требует CRUD для Tariff
5. **Контроль остатков расходников** — требует ConsumableNote

### Связи между сущностями

```
User 1:N House (owner_id)
User 1:N Booking (tenant_id)
House 1:N Booking
House 1:N ConsumableNote
Tariff используется в Booking (через guests_planned JSON)
```
