# Техническое видение проекта

## Технологии

**Python 3.12+** — основной язык.

**Управление зависимостями:**
- `uv` — установка зависимостей, виртуальное окружение
- `pyproject.toml` — метаданные проекта и список зависимостей

**Ключевые библиотеки:**
- `aiogram` — Telegram Bot API, polling
- `openai` — OpenAI-compatible клиент для LLM
- `pydantic` — валидация конфигурации и данных
- `python-dotenv` — загрузка переменных окружения

**Автоматизация:** `make` — команды install, run, lint, format

---

## Принципы разработки

- **KISS** — минимальная сложность, только необходимое для MVP
- **Один класс — один файл** — чёткая структура, легко ориентироваться
- **Явное лучше неявного** — конфигурация через env, нет хардкода секретов
- **Функциональная целостность** — бот работает из коробки после `make install && make run`

## Архитектура

**Паттерн:** Модульная структура с разделением ответственности

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   Telegram      │────▶│   aiogram       │────▶│   Handler       │
│   (пользователь)│     │   (polling)     │     │   (обработка)   │
└─────────────────┘     └─────────────────┘     └────────┬────────┘
                                                         │
                              ┌──────────────────────────┼──────────┐
                              ▼                          ▼          │
                        ┌─────────┐                ┌──────────┐     │
                        │  LLM    │◀───────────────│  Service │     │
                        │ Service │                │  Layer   │     │
                        └────┬────┘                └────┬─────┘     │
                             │                          │           │
                             ▼                          ▼           │
                        ┌─────────┐                ┌──────────┐     │
                        │RouterAI │                │ Booking  │─────┘
                        │   API   │                │  State   │
                        └─────────┘                └──────────┘
```

**Поток данных:**
1. Пользователь пишет в Telegram
2. aiogram получает update → передаёт в Handler
3. Handler извлекает контекст → вызывает LLM Service
4. LLM Service формирует промпт → запрашивает RouterAI
5. Ответ LLM парсится → обновляется Booking State
6. Пользователь получает ответ

## Модель данных / Состояние

**Booking (бронирование):**
```python
- id: str                    # UUID бронирования
- house: str                 # Название дома ("старый дом", "новый дом")
- check_in: date             # Дата заезда
- check_out: date            # Дата выезда
- guests: int                # Количество гостей
- created_by: int            # Telegram user_id
- created_at: datetime       # Время создания
- status: str                # "pending" | "confirmed" | "cancelled"
```

**Хранение:** In-memory (dict) для MVP, без БД. Состояние теряется при перезапуске — допустимо для MVP.

## Работа с LLM

**Провайдер:** [RouterAI](https://routerai.ru/) — OpenAI-compatible API

**Конфигурация:**
- `ROUTERAI_API_KEY` — ключ API
- `ROUTERAI_BASE_URL` — базовый URL API (по умолчанию: `https://routerai.ru/api/v1`)
- `LLM_MODEL` — модель (уточнить у провайдера)
- `SYSTEM_PROMPT` — системный промпт бота

**Клиент:** `openai` Python SDK с кастомным `base_url`

**Функции LLM:**
- Извлечение намерений из естественного языка
- Парсинг дат ("следующие выходные" → конкретные даты)
- Валидация запросов (доступность дома, вместимость)
- Генерация ответов пользователю

**Стратегия:** Zero-shot с системным промптом, без fine-tuning для MVP.

## Сценарии работы пользователя

**Ограничения:**
- Максимальное количество гостей в доме: **16 человек**
- При превышении — бронирование разрешено, но бот шутит

**Сценарий 1: Бронирование**
1. Пользователь: *"Забронируй старый дом на следующие выходные, 6 человек"*
2. Бот уточняет даты → проверяет пересечения → создаёт бронирование
3. Ответ: *"Готово! В старом доме с 15 по 17 марта будет 6 человек. Свободных мест осталось: 10"*

**Сценарий 2: Бронирование с перебором (юмор)**
1. Пользователь: *"Бронируем на майские, нас 20 человек"*
2. Бот: *"Ого, будет весело! Записал 20 человек на майские — будете как селёдки в бочке, но место ваше!"*

**Сценарий 3: Уточнение загруженности**
1. Пользователь: *"А сколько народу будет в доме 15 марта?"*
2. Бот суммирует все бронирования на эту дату → отвечает

**Сценарий 4: Перенос дат**
1. Пользователь: *"Перенеси наше бронирование с 15 на 22 марта"*
2. Бот идентифицирует бронирование → проверяет доступность → обновляет даты

**Сценарий 5: Отмена**
1. Пользователь: *"Отмени наше бронирование"*
2. Бот идентифицирует → подтверждает → меняет статус на cancelled

## Конфигурация и секреты

**Файл `.env`:**
```bash
# Telegram
TELEGRAM_BOT_TOKEN=your_bot_token_here

# RouterAI
ROUTERAI_API_KEY=your_routerai_key_here
ROUTERAI_BASE_URL=https://routerai.ru/api/v1
LLM_MODEL=...

# Bot
SYSTEM_PROMPT="Ты бот для бронирования загородного жилья..."
```

**`.env.example`** — шаблон с пустыми значениями в репозитории
**`.env`** — реальные значения, в `.gitignore`

## Логирование

- Стандартный `logging` из Python
- Уровень: `INFO` для продакшена, `DEBUG` для разработки
- Формат: `%(asctime)s - %(name)s - %(levelname)s - %(message)s`
- Вывод: stdout (Docker-friendly)

## Сборка и запуск

**Локально (без Docker):**
```bash
cp .env.example .env
# отредактировать .env
make install
make run
```

**Docker:**
```bash
docker build -t booking-bot .
docker run --env-file .env booking-bot
```

**Makefile:**
```makefile
install:
	uv sync

run:
	uv run python -m bot.main

lint:
	uv run ruff check .

format:
	uv run ruff format .

docker-build:
	docker build -t booking-bot .

docker-run:
	docker run --env-file .env booking-bot
```

## Деплой (Railway)

**Платформа:** [Railway](https://railway.app/)

**Настройка:**
1. Подключить GitHub репозиторий
2. Добавить переменные окружения в Railway Dashboard
3. Автодеплой при push в main

**Файлы:**
- `Dockerfile` — сборка образа
- `railway.json` (опционально) — конфигурация деплоя

**Процесс:**
- Railway автоматически собирает из Dockerfile
- Запускает контейнер с переменными окружения
- Предоставляет URL для health-check (если нужен)
